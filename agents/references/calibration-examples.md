# 評価キャリブレーション例

レビューエージェントが MUST / SHOULD / NICE の重要度を判定する際の境界例集。

## MUST の例（修正しなければバグ・規約違反・セキュリティリスクが発生）

| コード例 | カテゴリ | 理由 |
|---------|---------|------|
| ハードコードされた色値 in `プロジェクトのプレゼンテーション層（CLAUDE.md 参照）/` | デザインシステム | 規約: プロジェクトのデザインシステムクラス（CLAUDE.md 参照） 必須使用 |
| SDK を直接 import in `プロジェクトのプレゼンテーション層（CLAUDE.md 参照）/` | レイヤー違反 | 規約: SDK 直接アクセス禁止 |
| Service の戻り値が `Result` 型でない | エラーハンドリング | 規約: Result 型必須 |
| Service / Repository / UseCase にテストファイルなし | テスト欠落 | 規約: TDD 必須対象 |
| `dynamic` / `any` 型の安易な使用 | 型安全性 | 型の安全性が損なわれる |
| 空の catch ブロック | エラーハンドリング | 例外の握りつぶし = バグの隠蔽 |
| API キーのハードコード | セキュリティ | 機密情報の直接記述 |

## SHOULD の例（動くが保守性・可読性・パフォーマンスが低下）

| コード例 | カテゴリ | 理由 |
|---------|---------|------|
| テスト名が規約の命名パターンに沿っていない | テスト規約 | 推奨だがバグリスクなし |
| 3段以上のネスト | 簡潔性 | 可読性の問題だが動作に影響なし |
| テストが後付け（TDD 非準拠） | TDD | テスト自体は存在するが順序が逆 |
| 正常系のみのテスト（異常系なし） | テスト網羅性 | テストはあるが不完全 |
| 既存ユーティリティの再実装 | 再利用性 | 重複コードだが動作はする |

## NICE の例（SHOULD にしない — 主観的好みに近い）

| コード例 | カテゴリ | 理由 |
|---------|---------|------|
| 変数名がやや長い | 命名 | 主観的好み |
| 自明なコードにコメントがない | ドキュメント | コメント不要な場合も多い |
| early return にできるが if-else で書いている | 簡潔性 | 好みの範囲 |

## 判定の原則

1. **MUST のインフレーションを防ぐ**: 規約明記 or バグリスク or セキュリティリスクの場合のみ
2. **迷ったら SHOULD**: MUST か SHOULD か判断が難しい場合は SHOULD
3. **主観を排除**: 規約に根拠がない指摘は NICE 以下
4. **既存コードとの一貫性**: プロジェクト内で広く使われているパターンに合わせる
