# Updating the Harness

This guide explains how to update ai-dev-harness when new template versions are released.

## Overview

ai-dev-harness follows a **template-driven** architecture. Your project contains:

1. **`harness.yaml`** -- Your project-specific configuration (you own this)
2. **Generated files** -- CLAUDE.md, settings.json, etc. (generated from templates)
3. **Templates** -- The template files in `templates/` (provided by ai-dev-harness)

When updating, templates change but your `harness.yaml` stays the same. You re-run `init.sh` to regenerate the output files from updated templates.

---

## Update Process

### Step 1: Check your current version

Look at the `version` field in your `harness.yaml`:

```yaml
version: "1.0"
```

### Step 2: Pull the latest templates

If you installed ai-dev-harness via git clone:

```bash
cd path/to/ai-dev-harness
git pull origin main
```

If you installed it as a submodule:

```bash
git submodule update --remote ai-dev-harness
```

### Step 3: Check the changelog

Review `CHANGELOG.md` for breaking changes between your version and the latest:

```bash
cat CHANGELOG.md
```

Pay special attention to:
- **BREAKING** -- Changes that require `harness.yaml` modifications
- **NEW FIELD** -- New optional fields added to `harness.yaml`
- **DEPRECATED** -- Fields that will be removed in a future version

### Step 4: Update harness.yaml if needed

If the changelog mentions new required fields or breaking changes, update your `harness.yaml` accordingly. Use `harness.yaml.example` as a reference for the latest schema.

```bash
# Compare your config with the latest example
diff harness.yaml path/to/ai-dev-harness/harness.yaml.example
```

### Step 5: Regenerate configuration files

```bash
./init.sh
```

This regenerates all output files from the updated templates while preserving your `harness.yaml` settings.

### Step 6: Review the changes

```bash
git diff
```

Check that the regenerated files look correct. The diff should only show template improvements, not unintended changes to your configuration.

### Step 7: Commit

```bash
git add -A
git commit -m "chore: update ai-dev-harness templates to vX.Y"
```

## Handling Breaking Changes

### New required fields

If a new version adds a required field to `harness.yaml`:

1. The changelog will document the field, its purpose, and the default value
2. Add the field to your `harness.yaml` before running `init.sh`
3. If you skip this step, `init.sh` will print a warning with the recommended default

### Renamed fields

If a field is renamed:

1. The changelog will list the old and new names
2. Update your `harness.yaml` to use the new name
3. The old name may continue to work for one version (deprecated) before being removed

### Removed features

If a template feature is removed:

1. The changelog will explain why and suggest alternatives
2. Remove the corresponding fields from your `harness.yaml`
3. Clean up any generated files that are no longer produced

## Preserving Custom Modifications

### Files you should NOT modify directly

These files are regenerated by `init.sh` and your changes will be overwritten:

- `CLAUDE.md` (generated from `CLAUDE.md.tmpl`)
- `.claude/settings.json` (generated from `settings.json.tmpl`)
- Skill instruction files (generated from skill templates)

### Files you CAN safely modify

These files are only generated once (if they don't exist) and are never overwritten:

- `docs/plan/tasks.json` (your task data)
- `docs/plan/progress.json` (your progress data)
- `docs/plan/master_plan.md` (your project plan)
- `docs/plan/architecture.md` (your architecture doc)
- Convention files in `.claude/rules/` (your project rules)

### Custom CLAUDE.md content

If you need to add project-specific content to CLAUDE.md that isn't covered by the template:

1. **Preferred:** Add a design context section via `harness.yaml`:

```yaml
# In harness.yaml, design context is passed through to the template
```

2. **Alternative:** Create a `CLAUDE.md.local` file with your additions. Then configure `init.sh` to append it:

```bash
# In init.sh (if supported)
cat CLAUDE.md.local >> CLAUDE.md
```

3. **Last resort:** Modify CLAUDE.md directly, but be aware it will be overwritten on next `init.sh` run. Keep a backup.

## Troubleshooting

### `init.sh` fails with "unknown field"

Your `harness.yaml` contains a field that the current template version doesn't recognize. Check the changelog for renamed or removed fields.

### Generated files look different after update

This is expected when templates change. Review the diff carefully:

- **Content changes** in CLAUDE.md or skill files reflect template improvements
- **Structural changes** (new sections, reordered content) are intentional
- **Missing content** might indicate a breaking change -- check the changelog

### Skills behave differently after update

Skill behavior is defined by the skill instruction templates. If a skill's behavior changes:

1. Check the changelog for skill updates
2. Review the skill template diff
3. If the new behavior is unwanted, you can pin to a specific template version by checking out a specific git tag

### Reverting an update

If an update causes problems:

```bash
# Revert to the previous template version
cd path/to/ai-dev-harness
git checkout <previous-tag-or-commit>

# Regenerate with the old templates
cd your-project
./init.sh
```

## Automatic Update Checks

You can set up a reminder to check for updates:

```bash
# Add to your shell profile (.bashrc, .zshrc)
# Checks for updates once per week
ai_dev_harness_check_update() {
  local harness_dir="$HOME/ai-dev-harness"
  local last_check="$harness_dir/.last-update-check"
  local now=$(date +%s)
  local week=604800

  if [ -f "$last_check" ]; then
    local last=$(cat "$last_check")
    if [ $((now - last)) -lt $week ]; then
      return
    fi
  fi

  echo "$now" > "$last_check"
  cd "$harness_dir" && git fetch --quiet
  local behind=$(git rev-list HEAD..origin/main --count 2>/dev/null)
  if [ "$behind" -gt 0 ]; then
    echo "[ai-dev-harness] $behind new commits available. Run: cd $harness_dir && git pull"
  fi
}
```

## Contributing Updates

If you've made improvements to templates that could benefit other projects:

1. Fork the ai-dev-harness repository
2. Make your changes in a feature branch
3. Ensure backward compatibility (don't break existing `harness.yaml` files)
4. Add a changelog entry
5. Submit a pull request

See the project's CONTRIBUTING.md for detailed guidelines.
